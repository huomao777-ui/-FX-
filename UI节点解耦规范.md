# UI节点解耦规范

## 目标

本规范用于约束 UI 场景、UI 脚本、绑定器脚本的节点引用方式，避免因为调整节点层级、父节点名称、节点位置而导致整页功能失效。

## 核心原则

- 不要把整套 UI 逻辑硬编码到单一父节点路径下。
- 不要假设某个控件必须永远挂在固定父节点上。
- 允许策划或美术在 Godot 编辑器中调整节点位置、父子分组、容器结构。
- 脚本应优先通过导出 `NodePath`、分组节点、局部查找、组件绑定来获取引用。
- 旧结构需要兼容时，只能作为兜底回退，不能覆盖已经找到的新结构节点。

## 节点引用规则

- 优先使用 `@export var 某节点路径: NodePath` 暴露给编辑器配置。
- 如果节点属于一个独立功能块，应先定位功能根节点，再在根节点内部查找子节点。
- 查找顺序应为：
  1. 编辑器显式配置的 `NodePath`
  2. 新结构下的功能根节点
  3. 旧结构兼容路径
- 如果新结构已经找到节点，禁止再用旧路径二次覆盖成 `null` 或其他引用。
- 同一脚本内不要出现两套互相覆盖的 `_cache_nodes()` 逻辑。

## 场景结构规则

- 手机外框、状态栏、APP内容应拆成独立功能节点。
- 状态栏绑定逻辑只依赖“状态栏根节点”，不依赖完整手机页面路径。
- APP 主控制脚本只依赖“APP 根节点”或各功能子根节点，不依赖手机外框路径。
- 可拖动区域、滚动区域、弹窗区域、订单区域、K线区域应各自有独立根节点。
- 不要为了省事把所有交互节点都挂在一个总父节点下再写死绝对路径。

## 交互脚本规则

- 拖动、滑动、弹窗、按钮显示切换、列表生成删除，必须按功能拆分缓存节点。
- 一个功能块失效时，不应连带让其他功能块全部失效。
- 弹窗脚本只管理弹窗本身和它依赖的按钮，不直接耦合整个页面。
- 滑动列表脚本只依赖列表根节点、模板节点、删除按钮节点。
- 顶部状态栏脚本只依赖时间、电量、WiFi、流量等状态节点。
- 页面允许保留一个轻量主控脚本，但只能负责全局进入/退出状态、跨模块信号转发、模块注册，不承载具体按钮、拖动、K线、列表逻辑。
- K线、货币列表、开户面板等复杂功能必须挂在各自功能根节点上，互相通过明确方法或信号通信。

## 注释规范

- 关键交互代码必须写明“为什么存在”，而不是只描述“做了什么”。
- 需要注释的功能包括：拖动吸附、滑动删除、弹窗外点击关闭、K线清空但保留坐标轴、强平线参与价格轴缩放、选币后保持当前框、动态生成空白框。
- 配置驱动的表现必须注明配置来源，例如货币颜色、利率、点差率、默认杠杆来自 JSON。
- 当代码是为了兼容编辑器节点移动、旧节点结构或装饰节点不拦截输入时，必须在对应代码旁写清楚原因。
- 禁止留下无意义注释，例如“设置变量”“调用函数”；注释应帮助后续迁移和调试。

## 修改场景时的约束

- 调整节点层级前，先确认脚本是否使用了硬编码路径。
- 新增包装节点如 `APP`、`手机相关节点` 时，优先修改“功能根节点解析”，不要大面积写死新绝对路径。
- 场景改层级后，必须检查：
  - 拖动是否仍可用
  - 滑动容器是否仍可滚动
  - 按钮是否还能弹出弹窗
  - 弹窗外点击关闭是否仍可用
  - 订单/K线/货币框切换是否仍可用

## 推荐做法

- 为每个页面写一个“根节点解析函数”。
- 为每个功能块写一个“局部节点缓存函数”。
- 通过 `Panel`、`Control`、`Button`、`ScrollContainer` 等局部根节点向下查找，而不是从场景根开始写整条路径。
- 对节点改名、换父节点、换容器有高概率发生的界面，必须先做解耦再扩展功能。
- 复杂交互参数优先做成中文 `@export`，例如拖动阈值、吸附比例、换位延迟、选中按钮颜色、弹窗层级，方便在 Godot 检视器中调节。
- 涉及交易、市场、账户、货币配置的数据，优先读取 JSON 或 `GameDataManager` 下的系统脚本，UI 脚本只做展示和必要的界面状态缓存。
- 交易含义必须用统一术语写清楚，例如“做多=买入左侧货币、卖出右侧货币”，避免利差、盈亏、强平线方向各自理解。

## 本次外汇 UI 常见错误与预防

### 节点移动后功能整片失效

- 症状：重组 `APP`、`手机相关节点`、`货币种类` 后，拖动、滚动、弹窗、K线一起失效。
- 常见原因：脚本从旧父节点硬编码路径查找控件，或新旧路径缓存互相覆盖。
- 预防方式：每个功能根节点只缓存自己内部节点；跨模块通信通过方法或信号，例如货币列表调用 K线控制器的 `show_pair()`，不要直接改 K线内部子节点。
- 验收点：移动父节点后，至少检查货币框点击、加号弹窗、左滑删除、顶部遮挡拖动、K线切换、开户面板打开关闭。

### 中文节点名或脚本出现乱码

- 症状：Godot 报 `Invalid character`、函数名/节点名变成乱码，或 `find_descendant_by_name()` 找不到节点。
- 常见原因：文件编码被错误保存，或 PowerShell 输出乱码被误认为文件内容乱码后继续编辑。
- 预防方式：脚本和 `.tscn` 保持 UTF-8；排查时用 `Get-Content -Encoding UTF8`；不要把终端乱码复制回脚本。
- 验收点：用 Godot headless 加载场景和脚本，而不是只看终端输出。

### typed Array 和 Variant 警告被当成错误

- 症状：`Array` 传给 `Array[Dictionary]` 报错，或 `:=` 从 Variant 推断类型导致 Parser Error。
- 常见原因：JSON、`Dictionary.get()`、节点查找返回值都是 Variant，Godot 严格模式下不能随意推断。
- 预防方式：从 JSON 或 `Dictionary` 取值后显式写类型，例如 `var config: Dictionary = parsed as Dictionary`；函数参数需要通用数组时使用 `Array`，内部再逐项转换。
- 验收点：每次改完跑一次 Godot `--check-only`，尤其是 JSON 和配置相关代码。

### UI 状态只更新一半

- 症状：选中按钮只有文字变色，整颗按钮没有变；切换货币后 K线更新但当前持仓卡不清空；清空一侧货币后要切出去再切回来才刷新。
- 常见原因：状态刷新函数只改了一个 Label，或者刷新入口只在点击货币框时触发，没有在选币/清币后触发。
- 预防方式：每个功能块保留一个完整刷新函数，例如 `_refresh_button_states()`、`_refresh_position_panel()`、`_refresh_slot_views()`；所有状态变化都走同一刷新入口。
- 验收点：按钮、文本、图标、边框、操作面板、K线、订单卡要在同一次状态变化后同步正确。

### K线图坐标、虚线、强平线不同步

- 症状：右侧某条刻度没有虚线，强平线压扁 K线，空白货币对清空了时间轴，断轴标记浮在坐标栏外。
- 常见原因：网格线、右侧标签、K线价格映射、强平线价格映射使用了不同的坐标数据。
- 预防方式：价格轴数据统一由一个函数生成；K线、虚线、右侧标签、强平线都使用同一份 `price_axis`；空白图只清 K线和柱状图，保留时间轴、虚线和 `--` 坐标。
- 验收点：切换时分日周月、空白货币框、已开户货币框、强平线远离 K线时都要检查。

### 交易算法写在 UI 里且方向不一致

- 症状：熊猫元利率高于樱花币时，做多隔夜利差显示为负；点差成本写死 `0.05%`；强平线只用 `0.82` 或 `1.18` 粗算。
- 常见原因：UI 脚本脱离 `GameDataManager/TradingSystem` 和 `trading_config.json`，且没有统一“做多/做空”的货币含义。
- 预防方式：合约每手价值、点差、滑点、可用杠杆、强平阈值从 `资源/数据/交易/trading_config.json` 或 `TradingSystem` 读取；UI 只在缺少系统数据时做显示兜底。
- 验收点：做多左侧高息货币利差为正，做空为负；点差成本随手数变化；强平线随方向、杠杆、保证金变化。

### 拖动交互互相抢输入

- 症状：左滑删除覆盖了上下换位；已开户货币对无法拖动排序；松手后延迟定时器还触发换位。
- 常见原因：横向滑动和纵向拖动共用一个 `dragging` 状态，没有区分方向、阈值和取消令牌。
- 预防方式：拖动上下文保存起点、模式、方向和请求令牌；横向只处理左滑删除，纵向只处理同组换位；松手时取消未完成的换位请求。
- 验收点：已开户只能在已开户组内换位，未开户只能在未开户组内换位；未配完整的框左滑会回弹；已开户不能露出删除按钮。

### 浮层和选中框层级冲突

- 症状：选中的货币框压住开户面板，货币选择弹窗被列表覆盖，确认开仓按钮点击区域异常。
- 常见原因：选中框、行容器、弹窗、确认按钮都使用默认 `z_index`，或只改了视觉没有改交互层级。
- 预防方式：弹窗类控件设置明确的 `z_as_relative = false` 和高 `z_index`；列表行内的删除按钮放在行容器背后，货币框移动时露出。
- 验收点：开户面板、确认开仓、货币选择弹窗都能在任意选中框上方正常点击。

## 外汇交易 UI 代码规范补充

- 货币、平台、合约、利率、点差、杠杆档位、默认手数，不要散落在 `.gd` 中；优先放在 JSON 或系统脚本中。
- UI 脚本可以做显示兜底，但兜底值旁必须写清楚来源和用途，避免后续被误认为正式交易算法。
- 开仓成功后写入货币框的数据要包含订单展示需要的字段，例如手数、方向、杠杆、买入汇率、保证金、点差成本、隔夜利息、强平线。
- 已开仓状态、已配置未开仓状态、空白/不完整状态必须分开判断，不能只用一个 `configured` 变量表示全部。
- 颜色和交互手感参数应尽量导出为中文变量，不要为了微调反复改脚本常量。
- 涉及容器自动布局的动画要谨慎。`VBoxContainer`、`FlowContainer` 会接管子节点位置，如果要做真实换位动画，应改为手动行布局或使用临时占位节点。
- 装饰节点默认不应拦截鼠标事件；可点击按钮才使用 `MOUSE_FILTER_STOP`。
- 每次改 UI 逻辑后至少跑一次 Godot 解析检查；涉及交互时还要在编辑器中手测点击、拖动、滑动、弹窗关闭、切换周期。

## Git 提交规范补充

- 只要一个功能块完成并通过最基本的解析检查，就应尽快提交一次 git，避免后续大改时难以回退或定位回归点。
- 推荐提交粒度：
  - 一个明确交互修复
  - 一个独立面板的显示逻辑接入
  - 一次节点重构后的稳定版本
- 提交信息要写清楚“修了什么”和“影响范围”，例如 `fix: 恢复开户面板强平线与点差计算`、`refactor: 拆分货币列表与K线控制脚本`。
- 提交前至少确认：
  - Godot `--check-only` 通过
  - 本次修改涉及的关键交互已手测
  - 没有把用户刚改的场景节点误覆盖
- 如果仓库根目录存在类似 `.env.github` 的凭据文件，只作为提交/推送时的环境配置来源，不要把令牌写进脚本、场景、JSON、日志或规范示例中。
- 需要联网推送、拉取、访问 GitHub API 或依赖 VPN 的操作，开始前先和用户确认当前网络条件是否可用；本地改文档、本地提交、解析检查不需要为此打断工作。

## 禁止做法

- 禁止长期依赖 `手机进入状态/某节点/某子节点/...` 这种整条硬编码路径作为唯一入口。
- 禁止新结构找到节点后，再被旧兼容路径重新赋值覆盖。
- 禁止把场景结构调整和功能逻辑重构混在一次修改里一起粗暴处理。
- 禁止为了短期修复，把多个独立功能都绑定到同一个父节点假设上。
- 禁止把正式交易算法写成 UI 临时比例，例如长期使用 `rate * 0.82` 表示强平线。
- 禁止在按钮选中态只改文字颜色而不刷新按钮整体状态，除非设计明确要求。
- 禁止让两个交互方向共享同一个无区分拖动状态，例如左滑删除和上下排序混在一个判断里。

- `界面/场景/外汇应用/`
