# Godot `.tscn` 修改规范

这份文档用于约束 AI 或人工在项目中修改 Godot 文本场景文件时的方式，避免把一个局部 UI 改动扩散成整份场景的语法损坏。

## 适用范围

适用于所有 `.tscn`、`.tres`、`.theme` 等 Godot 文本资源，尤其是：

- 节点很多的 UI 场景
- 含大量中文节点名、中文 `text`、中文 `@export` 属性名的场景
- 已经存在弹窗、状态机、脚本绑定、资源引用的复合场景

## 核心原则

1. 把 `.tscn` 当作“结构化资源文件”，不要当普通纯文本随意整文件重写。
2. 优先修改最小范围；一次只改一组节点或一类属性。
3. 场景一旦出现解析错误，先止损，再定位，不要在坏文件上继续叠加修改。
4. 任何自动改动都要假设中文节点头、中文路径、中文字符串是高风险区域。

## 允许的修改方式

### 1. 低风险：纯 ASCII 定点替换

只在以下内容中做精确替换，且必须能唯一定位：

- `type="Panel"` -> `type="Button"`
- `custom_minimum_size = Vector2(...)`
- `theme_override_styles/...`
- `flat = true`
- `text = "<"`、`text = ">"` 这类纯 ASCII 文本
- `visible = false/true`
- `mouse_filter = ...`

要求：

- 只能替换明确的 ASCII 片段
- 不改同一段周边的中文节点名和 `parent="..."` 路径
- 替换后立即重新检查该片段上下文

### 2. 中风险：整段节点块替换

仅在以下前提满足时允许：

- 已经锁定一整段节点块的起止范围
- 确认起止范围外没有共享子节点或重复 `unique_id`
- 替换块中所有引号、空行、缩进、属性顺序都已核对

要求：

- 优先从底部向上替换，避免行号偏移
- 替换后必须立即检查新块是否留下残余旧行
- 不允许边替换边猜测

### 3. 高风险：整文件读写

默认禁止以下方式直接改 `.tscn`：

- `Get-Content` + `Set-Content`
- 任意脚本把 `.tscn` 先读成普通字符串，再整体写回
- 会自动写入 BOM 的保存方式
- 未确认编码的编辑器或命令行工具整文件覆盖

只有在以下条件同时满足时才允许：

- 已确认文件编码和写回编码完全一致
- 写回方式保证 `UTF-8 without BOM`
- 已有可恢复副本
- 改动仅限纯 ASCII 安全片段

## 明确禁止的操作

1. 不要修改 `.tscn` 里的中文节点头文本，除非明确知道该行完整原始字节。
2. 不要手改 `parent="..."` 里的中文路径。
3. 不要在终端显示乱码的情况下，直接把终端里看到的那行复制回文件。
4. 不要在场景已经报解析错误后，继续同时改布局、脚本、节点类型三件事。
5. 不要把 `.tmp` 临时文件默认视为干净底稿。

## 典型风险点

### 1. 节点头损坏

高风险行：

- `[node name="..." type="..." parent="..." unique_id=...]`

常见错误：

- `name` 的结束引号丢失
- `parent` 的结束引号丢失
- `type="..."` 被吞进 `name` 或 `parent`

表现：

- `Expected '=' after identifier`
- `Parse error` 卡在看似无关的下一行

### 2. 文本属性拆行

高风险行：

- `text = "..."`

常见错误：

- 变成：
  - `text = "18:38`
  - 下一行只有 `"`
- 中文文本尾引号丢失

表现：

- `Parse Error` 出现在当前行或下一行
- `Expected '['`、`Expected '='`、`Parse error`

### 3. BOM 问题

高风险现象：

- 文件开头字节变成 `EF BB BF`

表现：

- 第 1 行报 `Expected '['`

要求：

- `.tscn` 一律写成 `UTF-8 without BOM`

### 4. 字面量换行残留

错误示例：

- `theme_override_styles/normal = ...`r`n...`

表现：

- 一整行属性被拼成非法文本
- Godot 在该行附近直接解析失败

## 推荐流程

### 修改前

1. 先确认当前场景文件能打开。
2. 找到唯一基线文件，不同时操作多个恢复副本。
3. 先复制出一份工作副本，例如：
   - `xxx_recover_a.tscn`
   - `xxx_recover_b.tscn`
4. 记录要改的节点路径、类型、相关 `unique_id`。

### 修改时

1. 一次只改一个小目标。
2. 先做 ASCII 安全替换。
3. 每改完一步，立即检查：
   - 文件头是否仍以 `[gd_scene` 开头
   - 是否出现单独一行 `"`
   - 是否出现字面量 `` `r`n ``
   - 关键节点头是否还完整

### 修改后

1. 优先用 Godot 重新打开场景。
2. 如果打不开，只根据最新报错行附近修一个点。
3. 不要在未重新加载成功前继续做下一批视觉调整。

## 出错后的止损流程

如果场景已经打不开：

1. 立刻停止继续修改这份文件。
2. 保存当前坏文件副本，用于比对，不要覆盖唯一现场。
3. 从以下顺序寻找可恢复版本：
   - Git 已提交版本
   - `.tmp` 临时文件
   - Godot 编辑器缓存可定位的同名资源
4. 选一份“当前能打开”的文件作为新基线。
5. 之后只把需要的改动重新迁移过去，不在坏文件上补洞式修复。

## 恢复副本使用规范

如果使用 `.tmp` 文件恢复：

1. 不直接覆盖主场景。
2. 先另存为：
   - `xxx_recover_a.tscn`
   - `xxx_recover_b.tscn`
3. 分别验证能否打开。
4. 只有能打开的那份才允许继续改。

## 对 AI 修改 `.tscn` 的硬约束

AI 修改 `.tscn` 时必须遵守：

1. 优先使用最小范围的 ASCII 替换。
2. 禁止连续多轮整文件文本重写。
3. 任何一次写回后，如果场景打不开，下一轮只能做“定位和恢复”，不能继续做新需求。
4. 如果已经连续出现两类不同语法错误，必须停止直接改主场景，改为：
   - 恢复副本
   - 或转为编辑器内人工操作说明
5. 对 `.tscn` 的任何自动写回，都必须避免 BOM。

## 实战建议

对于按钮样式这类需求，优先顺序应为：

1. 可打开基线场景
2. 只改目标按钮组
3. 先改一个按钮验证
4. 成功后再批量改同类按钮

不要一上来同时做：

- 节点类型替换
- 子节点删除
- 样式资源新增
- 文本替换
- 底部按钮同步改造

## 当前项目的结论

对本项目而言，`财经报.tscn` / `财经报_recover_*.tscn` 这类场景已经证明属于高风险 `.tscn`：

- 中文节点名多
- 中文文本多
- 脚本导出属性多
- 弹窗结构复杂
- 容易因为文本写回方式错误而全文件损坏

因此后续修改必须遵循本规范，不再把它当普通文本文件直接反复整文件重写。
